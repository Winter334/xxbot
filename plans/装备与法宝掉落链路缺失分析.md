# 装备与法宝掉落链路缺失分析

## 1. 文档目的

本文用于记录当前项目中“装备掉落”“法宝掉落”相关链路的实现现状、已确认缺失点、证据位置与修复方向，作为后续补全掉落系统的依据。

本文中的“法宝”默认指装备系统中的 artifact 装备，而不是功法/技能物品。

---

## 2. 结论摘要

当前结论如下：

1. **装备/法宝系统本身已经实现**，包括生成、穿戴、培养、分解、查询与面板展示。
2. **装备/法宝实际掉落链路未完整接入玩法结算**。
3. 当前玩法结算中，和装备/法宝最接近的奖励形式是：
   - `equipment_score`
   - `artifact_score`
4. 这些奖励目前表现为**分数/待确认奖励摘要**，而不是实际 `EquipmentItem` 实例掉落。
5. **功法/技能掉落是完整接入的**，说明“特殊掉落物接入玩法结算”的技术路径是存在的，但该路径尚未扩展到装备/法宝。

一句话概括：

> 当前项目已经有完整的装备/法宝管理系统，但缺少“副本/战斗结算 -> 生成装备/法宝实例 -> 持久化 -> 记录掉落 -> 面板展示”的完整掉落链路。

- `src/application/equipment/equipment_service.py:331`
- `src/application/equipment/equipment_service.py:345`
- `src/application/equipment/equipment_service.py:358`

这说明：

- “生成一件装备/法宝实例”不是缺失能力。
- 缺的是“谁在玩法奖励结算时调用它”。

### 3.2 装备/法宝面板与管理链路已存在

装备面板查询服务已经接入：

- `src/application/equipment/panel_query_service.py:153`

- 当前问题不在“装备系统不存在”，而在“掉落入口没打通”。

### 3.3 Artifact（法宝）已经进入角色评分系统

Artifact 分数已在评分逻辑中作为独立部分处理：

- `src/domain/ranking/rules.py:137`

这说明：

- “法宝”在项目中是一个明确存在的装备类别。
- 但这里体现的是评分贡献，不代表法宝掉落链路已经实现。

## 9. 当前缺失点清单

当前至少存在以下缺失点。

### 9.1 缺少玩法层的装备掉落决策

当前没有明确逻辑决定：

- 什么时候掉装备；
- 什么时候掉法宝；
- 掉落槽位如何选；
- 品质如何选；
- 阶数如何选；
- 是否由楼层、节点类型、种子决定；
- `equipment_score` / `artifact_score` 与实际掉落之间是什么映射关系。

### 9.2 缺少“分数 -> 实际装备实例”的转换逻辑

当前奖励系统累计的是：

- `pending_equipment_score`
- `pending_artifact_score`

但未看到后续转换为：

- 一件或多件 `EquipmentItem`
- artifact 装备实例

因此目前这些字段更像中间态奖励值，而不是终态掉落。

### 9.3 缺少玩法结算中的装备实例持久化调用

虽然 `EquipmentService.generate_equipment(...)` 存在，但未看到结算流程中调用它，因此缺少：

- 实例生成；
- 入库；
- 与当前角色关联；
- 与结算记录关联。

### 9.4 缺少 `DropRecord.items_json` 的装备/法宝实例协议

当前 `DropRecord` 对无尽主要记录：

- 稳定资源摘要；
- 待确认奖励摘要；
- 技能掉落摘要。

缺少用于装备/法宝实例的标准结构，例如可能需要包含：

- `entry_type`
- `equipment_item_id`
- `slot_id`
- `is_artifact`
- `quality_id`
- `rank_id`
- `template_id`
- `display_name`
- `source_record_id`

### 9.5 缺少面板层的装备/法宝掉落展示逻辑

当前面板摘要构建函数不理解装备/法宝实例条目，因此缺少：

- 最近掉落文案；
- 详细掉落摘要；
- 与技能掉落类似的结构化显示。

### 9.6 缺少集成测试覆盖

当前测试只验证：

- `equipment_score` / `artifact_score` 的结算值；
- `skill_drop` 的存在。

缺少以下测试：

1. 无尽撤退结算产出装备实例。
2. 无尽战败结算产出装备实例或法宝实例。
3. `DropRecord.items_json` 记录具体装备实例信息。
4. 装备面板最近掉落摘要可正确显示装备/法宝掉落。
5. 掉落后仓储中确实新增对应装备实例。

## 10. 建议修复顺序

以下为建议修复顺序，仅用于指导实现拆分，不约束具体代码结构。

### 第一步：定义装备/法宝掉落语义

先明确下面这些产品规则：

1. `equipment_score` 是否继续保留。
2. `artifact_score` 是否继续保留。
3. 分数是否用于决定：
   - 掉落数量；
   - 掉落品质；
   - 掉落阶数；
   - 是否保底法宝。
4. 不同节点类型（普通/精英/锚点首领）如何影响实际掉落。

如果这一步不先明确，后续实现容易出现“继续累计分数，但没人消费”的问题。

### 第二步：补一个玩法可调用的装备掉落应用层入口

建议参考功法掉落模式，增加一层专门面向玩法结算的应用服务，负责：

- 读取当前奖励上下文；
- 决定是否掉装备 / 法宝；
- 调用 `EquipmentService.generate_equipment(...)`；
- 生成可写入 `DropRecord` 的掉落摘要；
- 返回给结算层用于拼装 `final_drop_list`。

### 第三步：把无尽结算接上装备/法宝掉落

重点接入位置应首先关注：

- `src/application/dungeon/endless_service.py:955`
- `src/application/dungeon/endless_service.py:982`

目标是让无尽结算最终能够同时返回：

- 资源奖励；
- 装备/法宝实例掉落；
- 功法掉落。

### 第四步：扩展 `DropRecord.items_json` 协议

为装备/法宝增加标准条目结构，确保：

- 可以审计；
- 可以回放；
- 可以在面板显示；
- 后续如果要做“查看最近掉落详情”时不用返工。

### 第五步：扩展面板摘要展示

优先改造：

- `src/application/equipment/panel_query_service.py:184`

使其能够显示具体装备/法宝，例如：

- 装备名
- 品质
- 槽位
- 是否法宝
- 关键属性摘要

### 第六步：补集成测试

优先补以下测试用例：

1. 无尽撤退结算后新增装备实例。
2. 锚点首领结算后可新增 artifact 实例。
3. `DropRecord` 中存在 `equipment_drop` / `artifact_drop` 条目。
4. 最近掉落摘要能显示具体装备/法宝信息。
5. 重复读取结算结果不重复生成装备。

## 11. 修复时需要特别注意的风险

### 11.1 避免重复结算重复生成装备

当前无尽结算已有“重复读取不重复入账”的约束，装备/法宝掉落接入后必须沿用这一原则。

否则会出现：

- 重复打开结算面板，重复领装备；
- 重新读取结算结果，重复插入装备实例。

### 11.2 避免只写掉落摘要、不写真实实例

如果只在 `final_drop_list` 里追加一个文本摘要，但没有真实调用装备生成并持久化，那仍然不算掉落链路完成。

完整标准应当是：

1. 生成实际装备实例；
2. 写入仓储；
3. 写入掉落记录；
4. 面板可见；
5. 测试可验证。

### 11.3 避免玩法层直接拼装复杂装备细节

玩法层更适合决定“掉什么”，而不是手工拼装备实例结构。

建议仍由装备系统负责：

- 实例生成；
- 命名；
- 模板解析；
- 随机 affix；
- 持久化。

这样可以减少玩法层和装备系统的耦合。

## 12. 最终判断

截至当前代码状态，可以给出以下判断：

### 12.1 装备掉落

**未实现完整。**

原因：

- 有装备生成能力；
- 有装备系统与面板；

- 当前无尽结算只产出 `equipment_score`，不是装备实例。

### 12.2 法宝掉落（artifact 装备）

**未实现完整。**

原因：

- 有 artifact 装备概念；
- 有 artifact 评分与培养；
- 但未发现玩法结算实际生成 artifact 实例；
- 当前无尽结算只产出 `artifact_score`，不是法宝实例。

### 12.3 功法掉落

**已实现完整。**

原因：

- 有配置；
- 有实例生成；
- 有持久化；
- 有无尽结算接入；
- 有 `DropRecord`；
- 有集成测试。
